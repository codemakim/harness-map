# harness-map v0.1 Design

## Product

`harness-map` is a local TypeScript CLI that shows what instruction files an
agent reads for a given file path.

The first version supports Codex only. It does not call AI services, does not
call the network, and does not try to grade harness quality.

## Positioning

Core question:

> At this path, what does this agent actually read?

This is different from linting. Linting says whether a harness is good or bad.
Mapping says which files apply, in which order, with which size impact and
obvious contradictions.

## Implementation Language

Use TypeScript on Node.js.

Reasons:

- Natural `npx harness-map ...` distribution.
- Built-in file system APIs cover the v0.1 needs.
- JSON output and terminal formatting are simple.
- Most target projects already have Node available.
- Rust or Go can come later only if startup time or standalone binaries matter.

## Commands

### `harness-map explain <file>`

Shows the effective instruction chain for one file.

Default agent: `codex`.

Options:

- `--agent codex`
- `--json`
- `--cwd <dir>`

By default, `explain <file>` simulates launching Codex from the target file's
directory. `--cwd` overrides that endpoint when the user wants to inspect the
instructions for a different working directory.

Output includes:

- Target file path
- Agent name
- Ordered instruction files
- Per-file size
- Effective size
- Budget
- Override summary
- Warnings

### `harness-map tree`

Shows discovered Codex instruction files under the current workspace.
In Git repositories, ignored untracked files are excluded.

Useful before asking about a specific file.

### `harness-map budget`

Shows total discovered instruction size and flags files or paths likely to hit
the Codex instruction budget.

### `harness-map doctor`

Runs repository-level checks:

- Missing referenced files
- Missing documented package scripts
- Instruction files over budget by themselves

## Deferred Commands

### `harness-map compare <file> --agents codex,claude,cursor`

Deferred until a second adapter exists.

Reason: v0.1 should prove path-specific reconstruction before adding adapter
drift reporting.

## Codex Discovery Rules

For a target file:

1. Resolve `cwd`.
2. Resolve target file path relative to `cwd`.
3. Collect global Codex instructions from `~/.codex/AGENTS.override.md` if
   present, otherwise `~/.codex/AGENTS.md` if present.
4. Walk from the project root to the simulated Codex working directory.
5. At each directory, include at most one instruction file.
6. Prefer `AGENTS.override.md` over `AGENTS.md` in the same directory.
7. Apply parent directories before child directories.

The v0.1 model treats `AGENTS.override.md` as a replacement for the same
directory's regular `AGENTS.md`, not as an extra file appended beside it.

## Budget

Default Codex instruction budget: `32 KiB`.

Calculate size as UTF-8 bytes of included instruction file contents. Present
human-readable KiB with one decimal place.

The CLI reports:

- `effectiveBytes`
- `budgetBytes`
- `effectiveSize`
- `budgetSize`
- `overBudget`

## Override Detection

v0.1 keeps override detection intentionally small.

Detect only simple replacements in later files:

- Package manager mentions: `npm`, `pnpm`, `yarn`, `bun`
- Test command mentions: commands containing `test`

Example:

```text
Overrides:
- Package manager: npm -> pnpm
- Test command: npm test -> pnpm test:web
```

This is a helpful summary, not a full semantic diff.

## Warning Detection

### Missing Referenced Files

Scan instruction markdown for local relative references:

- Markdown links: `[text](docs/file.md)`
- Inline paths containing `/` and ending in `.md`, `.mdx`, `.json`, `.yml`,
  `.yaml`, `.toml`

Ignore:

- URLs
- Anchor links
- Home-relative paths such as `~/.codex/config.toml`
- Template placeholders and agent import syntax
- Bare inline filenames; use a Markdown link when existence should be checked
- Absolute paths outside the workspace

Resolve relative references from the instruction file directory, then the
project root. Treat either existing location as valid because nested agent
files commonly document repository-root paths.

### Missing Package Scripts

Scan instruction markdown for `npm run <script>` and `pnpm <script>` style
commands.

Check nearest `package.json` from the instruction file directory upward.

For commands written as `cd <dir> && npm run <script>`, start from `<dir>`.

Warn when the referenced script is not defined in `scripts`.

## JSON Output

`--json` returns a stable object:

```json
{
  "agent": "codex",
  "target": "apps/web/src/pages/Home.tsx",
  "cwd": "/repo",
  "budgetBytes": 32768,
  "effectiveBytes": 9216,
  "overBudget": false,
  "instructions": [
    {
      "path": "/home/user/.codex/AGENTS.md",
      "displayPath": "~/.codex/AGENTS.md",
      "bytes": 1228,
      "kind": "global",
      "precedence": 1
    }
  ],
  "overrides": [
    {
      "kind": "package-manager",
      "from": "npm",
      "to": "pnpm",
      "source": "apps/web/AGENTS.md"
    }
  ],
  "warnings": [
    {
      "kind": "missing-reference",
      "message": "docs/frontend-conventions.md does not exist",
      "source": "apps/web/AGENTS.md"
    }
  ]
}
```

## Internal Shape

Keep the first implementation boring:

```text
src/
  cli.ts
  codex.ts
  inspect.ts
  package-json.ts
  output.ts
```

No package split in v0.1. The future `packages/core` and adapter packages are
documented but not created until a second adapter exists.

## Test Plan

Small fixture-based tests are enough:

- Nested `AGENTS.md` files apply parent to child.
- `AGENTS.override.md` replaces same-directory `AGENTS.md`.
- Missing markdown reference emits a warning.
- Missing package script emits a warning.
- `--json` emits parseable JSON with stable keys.

No snapshot-heavy test suite for v0.1.

## Non-Goals

- No AI calls.
- No network calls.
- No scoring.
- No security audit.
- No Claude, Cursor, Copilot adapter.
- No monorepo package split.
- No web UI.

## Open Source Hook

The first README should sell the debugging story:

> Agents do not read one prompt. They read a harness. `harness-map` shows the
> harness that actually reaches a file.

That line is enough for v0.1. More branding can wait until the CLI works.
